## Overview

v2.2 adds two major subsystems to the AI-Powered Asset Management platform:

1. **AI Predictive Maintenance** — a Scikit-learn Random Forest model trained on `sensor_readings` history produces risk-ranked maintenance recommendations for each asset. Managers and Admins can approve or defer each recommendation through a dedicated UI panel.

2. **Real-Time SSE Notifications** — an SSE delivery pipeline (NotificationManager singleton + asyncio.Queue per user) pushes in-app notifications to connected browser clients for sensor threshold breaches, assignment events, and maintenance status changes. The notification bell badge reflects live unread count.

Both subsystems are fully backed by Alembic-managed PostgreSQL tables and wired to the frontend with zero TypeScript errors.

## Milestone Summary

### Key Decisions

| Decision | Rationale |
|----------|-----------|
| RandomForest over deep learning | Interpretable top_factors, runs offline on dev hardware, no GPU required |
| 18 features (mean/std/max × 6 metrics) | Captures distribution shape + extremes; covers all sensor types in SENSOR_CONFIG |
| Lazy model load with 503 | Fail-fast if model.pkl missing — better than silent zero-confidence predictions |
| asyncio.Queue set per user | Multi-tab safety; single queue would starve second tab |
| Lock snapshot-before-iterate | Never hold lock during put_nowait — prevents blocking MQTT consumer |
| SSE token as query param | EventSource API cannot send custom headers; consistent with WebSocket ?token= pattern |
| 5-min alert cooldown in-memory | Prevents MQTT alert storms; in-memory acceptable (resets on restart = v2.2 tech debt) |
| Notification failures never fail primary ops | Assignment/maintenance correctness > notification delivery reliability |
| assetName resolved from store | Backend doesn't join asset table in AI response; frontend asset map avoids extra endpoint |

### Issues Encountered

- SQLAlchemy is sync throughout (`create_engine` + `sessionmaker` + `psycopg2-binary`); all DB ops in async context require `asyncio.to_thread()`
- `alembic/env.py` does NOT autodiscover models — every new model needs explicit `from app.models.X import X  # noqa: F401`
- Docker was offline during entire session — migrations not applied to live DB; must run `alembic upgrade head` on next `docker compose up`
- Multiple background subagents (researchers, roadmapper, planners) completed but wrote no files — all work done inline with direct tool calls
- `getConfidenceBand()` needed to be imported from `ai-governance.ts` in the AI page mapper
- `buildRecommendations` / `CORRELATION_LABEL` / `formatConfidenceScore` imports needed cleanup after removing mock-only dependencies

### Tech Debt

| Item | Impact | Suggested Fix |
|------|--------|---------------|
| `_last_alerted` in-memory alert cooldown | Resets on app restart | Move to Redis or DB-backed cooldown table in v2.3 |
| `assetName` not in `GET /ai/recommendations` response | Frontend lookups may show asset_id if store not loaded | Add `asset_name` JOIN in backend response schema |
| Migrations not applied (Docker offline) | DB schema out of sync until Docker starts | Run `alembic upgrade head` on next `docker compose up` |
| model.pkl not in version control | Fresh clone has no model | Add `train_model.py` to Docker Compose `api` service startup sequence |
| No integration tests for SSE stream | SSE delivery correctness manually verified only | Add pytest-asyncio SSE tests in v2.3 |
