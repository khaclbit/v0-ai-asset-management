### §1.1 Entity Notes — Non-Obvious Design Decisions

Produce a Markdown table with columns: Entity | Field | Design Decision | Why Non-Obvious.
Include exactly these 10 rows (content per RESEARCH.md § "Non-Obvious Design Decisions"):

Row 1 — Users / is_active:
  Entity: Users
  Field: is_active
  Design Decision: Soft-delete only — set is_active = false; never hard-delete a User record
  Why Non-Obvious: Hard deletion would break AuditEvents.actor_id FK references for every
  historical audit event performed by that user

Row 2 — Users / hashed_password:
  Entity: Users
  Field: hashed_password
  Design Decision: bcrypt hash stored; plaintext password NEVER stored
  Why Non-Obvious: Security requirement — field name signals intent but value must always be
  the bcrypt output, never the raw input

Row 3 — Categories / sensor_types:
  Entity: Categories
  Field: sensor_types (JSON)
  Design Decision: JSON array of active sensor type strings; not a separate join table
  Why Non-Obvious: Categories are seeded at startup (5-10 rows, rarely changed) — JSON
  denormalization is acceptable and simpler; IoT Simulator reads this field to configure
  which sensors to activate per asset category

Row 4 — Assets / sensor_device_id:
  Entity: Assets
  Field: sensor_device_id
  Design Decision: Plain string (not a UUID FK) — maps to the IoT Simulator's device alias
  Why Non-Obvious: Looks like a missing FK to a Devices table; it is intentionally a string
  because the simulator uses an alias format (e.g. ASSET-001) that may not match the
  PostgreSQL UUID. Nullable — assets without IoT monitoring leave this blank.

Row 5 — Assignments / status (overdue):
  Entity: Assignments
  Field: status enum
  Design Decision: The `overdue` value is derived at query time; it is NOT stored persistently
  Why Non-Obvious: Frontend type system includes `overdue` as an AssignmentStatus enum value,
  creating the illusion it is stored. Backend computes: status = active AND
  expected_return_date < CURRENT_DATE. The database never contains a stored `overdue` value.

Row 6 — MaintenanceRecords / correlation_id:
  Entity: MaintenanceRecords
  Field: correlation_id
  Design Decision: String field carrying the AIRecommendations.id that triggered this record
  (when AI-triggered); NOT a hard UUID FK to AIRecommendations
  Why Non-Obvious: Looks like a missing FK — deliberately avoided to prevent coupling the
  Maintenance module to the AI module at the database FK level. Traceability is preserved via
  string equality, not a constraint.

Row 7 — SensorReadings / message_id:
  Entity: SensorReadings
  Field: message_id
  Design Decision: Carries a UNIQUE constraint — serves as the MQTT deduplication key
  Why Non-Obvious: MQTT QoS 1 guarantees at-least-once delivery; the same message can arrive
  twice on broker reconnect. Without this constraint, duplicate readings corrupt the rolling
  averages used by AI feature engineering.

Row 8 — AIRecommendations (whole entity):
  Entity: AIRecommendations
  Field: (whole entity)
  Design Decision: Only the Predictive ML service (app/ai/) may INSERT rows; all other
  modules are read-only consumers. Enforced at API middleware, not at DB constraint level.
  Why Non-Obvious: Nothing on the schema itself signals this write restriction

Row 9 — AIRecommendations / sla_deadline:
  Entity: AIRecommendations
  Field: sla_deadline
  Design Decision: Deadline datetime for Manager action on high-risk items; displayed as a
  countdown in the Manager UI; an escalation notification fires on breach
  Why Non-Obvious: Unusual to have an SLA field on a business data table

Row 10 — AuditEvents (whole entity):
  Entity: AuditEvents
  Field: (whole entity)
  Design Decision: Append-only — no UPDATE or DELETE operations ever. Corrections must be
  modelled as new events. AuditService exposes only an append() method.
  Why Non-Obvious: Standard tables allow full CRUD; this table behaves like an immutable ledger

## §2 API Module Overview

<!-- REQ:DATA-02 -->

### Table 1 — API Module Catalogue

Produce a Markdown table with columns: # | Module | Responsibility.
Nine rows, in this order, with these exact one-line responsibility statements:

1 | Auth
  Responsibility: Issues JWT access tokens on credential verification, validates tokens on every
  inbound request, and enforces role-based route guards for all nine API modules via FastAPI
  Depends() injection.

2 | Assets
  Responsibility: Manages the full asset lifecycle — creation, field editing, category
  association, and state transitions through the five-state machine (registered → available →
  assigned → maintenance → retired) including IoT device linkage via sensor_device_id.

3 | Assignments
  Responsibility: Orchestrates the request-approve-return workflow for asset loans, enforcing
  the five-state lifecycle (requested → active → overdue [derived] → closed → rejected) and
  recording the approving Manager/Admin on each approval event.

4 | Maintenance
  Responsibility: Manages maintenance record creation and state transitions
  (scheduled → in_progress → completed / blocked), including the AI-triggered ticket creation
  path invoked when a Manager approves an AI recommendation.

5 | IoT Telemetry
  Responsibility: Subscribes to the Mosquitto MQTT broker on the assets/+/sensors/+ topic,
  deserializes and deduplicates sensor payloads via message_id unique constraint, persists
  readings to sensor_readings, and broadcasts live updates to connected WebSocket clients.

6 | AI Predictions
  Responsibility: Reads historical sensor data from PostgreSQL, engineers the 9-feature vector
  per asset, runs inference through the pre-trained Scikit-learn/XGBoost model, and writes
  health_score, failure_risk, confidence, and top_factors exclusively to ai_recommendations
  (write-boundary enforced — no other table is written by this module).

7 | Notifications
  Responsibility: Aggregates trigger events from four source modules (AI Predictions, Assets,
  Maintenance, Assignments), creates per-user notification records for the four event types
  (high_failure_risk, warranty_expiry_warning, upcoming_maintenance, overdue_return), and
  delivers them to connected clients via Server-Sent Events (SSE).

8 | Audit
  Responsibility: Provides an append-only service that records every business, security, and
  AI-assisted event to audit_events with before/after state snapshots and correlation IDs —
  no UPDATE or DELETE capability is exposed by this module under any circumstances.

9 | Users
  Responsibility: Handles user account management — creation, role assignment (admin / manager /
  staff), profile updates, and soft-deactivation (is_active = false) — accessible only to users
  holding the Administrator role.
