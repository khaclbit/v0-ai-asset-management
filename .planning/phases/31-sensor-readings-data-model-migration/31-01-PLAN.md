  <name>Task T03: Create Alembic migration 0002_sensor_readings.py</name>
  <files>backend/alembic/versions/0002_sensor_readings.py</files>
  <action>
Create `backend/alembic/versions/0002_sensor_readings.py` following the exact structure of
`backend/alembic/versions/0001_initial.py` (read it first to match header format exactly).

**File header (verbatim):**
```
"""add sensor_readings table

Revision ID: 0002
Revises: 0001
Create Date: 2026-07-05 00:00:00.000000

"""
from typing import Sequence, Union

import sqlalchemy as sa
from alembic import op
from sqlalchemy.dialects import postgresql
```

**Revision identifiers:**
```python
revision: str = "0002"
down_revision: Union[str, None] = "0001"
branch_labels: Union[str, Sequence[str], None] = None
depends_on: Union[str, Sequence[str], None] = None
```

CRITICAL: `down_revision` MUST be `"0001"` — NOT `None`. Setting it to `None` creates two
heads and breaks `alembic upgrade head` (Pitfall 2 in RESEARCH.md).

**upgrade() function** — creates the table and three indexes:

op.create_table("sensor_readings",
    sa.Column("id", postgresql.UUID(as_uuid=True),
              server_default=sa.text("gen_random_uuid()"), nullable=False),
    sa.Column("device_id", sa.String(length=100), nullable=False),
    sa.Column("asset_id", sa.String(length=100), nullable=True),   # IOT-DB-03: no FK
    sa.Column("metric", sa.String(length=50), nullable=False),
    sa.Column("value", sa.Float(), nullable=False),
    sa.Column("unit", sa.String(length=20), nullable=False),
    sa.Column("recorded_at", sa.DateTime(timezone=True), nullable=False),
    sa.PrimaryKeyConstraint("id"),
)

# Primary composite index (IOT-DB-02): device+metric+time descending
op.create_index(
    "ix_sensor_readings_device_metric_recorded",
    "sensor_readings",
    ["device_id", "metric", sa.text("recorded_at DESC")],
)

# Secondary index: fast single-device fan-out (all metrics for a device)
op.create_index(
    "ix_sensor_readings_device_id",
    "sensor_readings",
    ["device_id"],
)

# Tertiary index: global time queries (retention cleanup, admin range scans)
op.create_index(
    "ix_sensor_readings_recorded_at",
    "sensor_readings",
    ["recorded_at"],
)

The `sa.text("recorded_at DESC")` as the third element of the composite index is the
correct Alembic 1.14.0 syntax for a descending column in an index (verified in RESEARCH.md
Assumption A3). If this syntax is rejected at runtime, fallback: pass
`postgresql_ops={"recorded_at": "DESC NULLS LAST"}` as a keyword argument to create_index.

**downgrade() function** — drops indexes in reverse creation order, then drops the table:

op.drop_index("ix_sensor_readings_recorded_at", table_name="sensor_readings")
op.drop_index("ix_sensor_readings_device_id", table_name="sensor_readings")
op.drop_index("ix_sensor_readings_device_metric_recorded", table_name="sensor_readings")
op.drop_table("sensor_readings")

Do NOT use `sa.Enum` anywhere in this migration (Pitfall 3 in RESEARCH.md).
Do NOT add ForeignKeyConstraint for asset_id (IOT-DB-03).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Migration runner → PostgreSQL | Alembic DDL statements executed as the `postgres` superuser inside Docker; malformed SQL can corrupt schema |
| env.py import → model module | If import path is wrong, `target_metadata` silently excludes `sensor_readings`, causing autogenerate to DROP the table in a future run |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-31-01 | Tampering | `0002_sensor_readings.py` — `down_revision` | mitigate | Task T03 verify step asserts `"0001"` is present; `alembic history` shows linear chain before running upgrade |
| T-31-02 | Tampering | `0002_sensor_readings.py` — accidental `ForeignKeyConstraint` | mitigate | Task T03 verify step asserts `ForeignKeyConstraint` is absent; SC-3 confirms at DB level |
