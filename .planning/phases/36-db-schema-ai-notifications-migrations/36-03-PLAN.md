      docker compose exec db psql -U postgres -d asset_management -c "\dt"
      Expected: rows for "ai_recommendations" and "notifications" are present.

    Step 4 — Inspect ai_recommendations schema:
      docker compose exec db psql -U postgres -d asset_management -c "\d ai_recommendations"
      Confirm all 14 columns visible:
        id, asset_id, recommendation, confidence, risk_level, risk_score,
        top_factors, correlation_id, approved_by, approved_at, action_state,
        defer_reason, sla_due_at, created_at
      Confirm FK constraints:
        fk_ai_recommendations_asset_id → assets.id (ON DELETE CASCADE)
        fk_ai_recommendations_approved_by → users.id (ON DELETE SET NULL)
      Confirm index: ix_ai_recommendations_asset_id

    Step 5 — Inspect notifications schema:
      docker compose exec db psql -U postgres -d asset_management -c "\d notifications"
      Confirm all 8 columns visible:
        id, user_id, type, title, message, is_read, href, created_at
      Confirm FK constraint:
        fk_notifications_user_id → users.id (ON DELETE CASCADE)
      Confirm index: ix_notifications_user_id_created_at

    Step 6 — Verify reversibility (downgrade cycle):
      docker compose exec api alembic downgrade -1
      # alembic current should now show 0003
      docker compose exec api alembic downgrade -1
      # alembic current should now show 0002
      docker compose exec api alembic upgrade head
      # alembic current should show 0004 (head) again
      # \dt should show both tables restored

    If `alembic upgrade head` in Step 1 fails with an error mentioning "multiple heads"

    If `alembic downgrade -1` errors, check for cascading FK data in dependent tables —
    downgrade tests should be run against a clean DB (no seeded data required in this phase).
